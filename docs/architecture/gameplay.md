## 1. Executive Summary

**What this system does:**
`astraweave-gameplay` is a single crate holding high-level, mostly-CPU game mechanics: melee combat (a physics raycast sweep plus an ECS combo-chain ticker), crafting/recipes, quests, branching dialogue, character stats, inventory/items, harvesting/biome-spawn, cutscene cue playback, the in-crate "weave op → Director plan" glue, swimming/oxygen "water movement" logic, and the Veilweaver vertical-slice cell metadata + tutorial state machine.

**Why it exists:**
It is the gameplay-logic layer that sits above the engine foundation (ECS, core, physics, nav, scene, cinematics) and below the game-specific binaries (examples and `veilweaver_slice_runtime`). It provides reusable mechanic types and a few ECS systems.

**Where it primarily lives:**
- `astraweave-gameplay/src/` — all modules (combat, combat_physics, crafting, quests, dialogue, stats, items, harvesting, biome, biome_spawn, cutscenes, weaving, weave_portals, weave_telemetry, water_movement, veilweaver_slice, veilweaver_tutorial, ecs).

**Status note:**
The crate is **not** a single integrated pipeline. It is a set of independent modules with very different wiring statuses. Three honesty signals an agent must internalize up front:
1. **Several gameplay TYPES are wired** into the runtime indirectly: `astraweave-ui` reads `QuestLog`/`Stats`/`Inventory`/`RecipeBook`; `astraweave-terrain` uses `BiomeRule`/`ResourceNode`/`spawn_resources`; `astraweave-audio` uses `Dialogue`/`DialogueState`; `veilweaver_slice_runtime` uses the `veilweaver_tutorial` ECS systems.
2. **The headline behavioral entry points are example-only or runtime-crate-only.** `perform_attack_sweep` (combat_physics), `apply_weave_op` (weaving), and the `CombatPlugin`/`CraftingPlugin`/`QuestPlugin` ECS plugins have **zero** non-test/non-example call sites inside the `astraweave-*` engine crates (verified §4/§5). They are exercised by `examples/` binaries and tests.
3. **`water_movement.rs` (swimming/oxygen/wet-status) is dormant** — it has zero callers anywhere in the workspace (examples included), and originates from an explicitly abandoned Enshrouded-style water plan (§7). It is **not** the W-series render-side water deformation system, and it does **not** consume any `WaterQuery`/fluids facade (§4, §6).

## 2. Authoritative Pipeline

This crate has no single end-to-end pipeline. It has several short, independent flows. The three most load-bearing are traced below. Each is a *capability the crate offers*; the production driver of each lives outside the crate (an example or the slice-runtime).

### 2.A Melee combat (two parallel paths)

```text
[Path A — physics sweep]                          [Path B — ECS combo ticker]
caller (example/test)                             App scheduler ("simulation" stage)
    │ perform_attack_sweep(phys, self_id,             │ combat_system(world)
    │   from, to, radius, dmg, dtype, &mut targets)   │   (registered by CombatPlugin)
    ▼                                                  ▼
combat_physics.rs:36                               ecs.rs:89
  role: raycast from `from`+y-offset toward `to`,    role: per-tick advance CAttackState
        cone-filter (dot>0.5), parry/iframe gate,          combo, Manhattan-distance reach
        then Stats::apply_damage                            check vs CTarget, mutate CHealth
    │                                                  │
    ▼                                                  ▼
Option<HitResult{target,damage,parried}>           CHealth.hp mutated in-place
```

`Path A` and `Path B` are **two different combat implementations in the same crate** that do not share code: A is a Rapier raycast against `PhysicsWorld` colliders returning a `HitResult`; B is a pure-ECS combo-chain timing/reach ticker mutating `CHealth`. See §6 (coexisting abstractions).

#### Stage detail — Path A
**File:** `astraweave-gameplay/src/combat_physics.rs:36-152`
**Role:** One-shot attack sweep. Casts a ray (`phys.query_pipeline.cast_ray_and_get_normal`, line 74) from `from + (0,1,0)` along the normalized `to-from` direction, excludes the attacker via `QueryFilter`, maps the hit collider back to a body id (`phys.id_of`, line 86), applies a ~60° forward cone filter (`dot < 0.5` reject, line 108), then resolves parry (consume on hit, lines 115-125) → iframe (block without consume, lines 128-136) → `target.stats.apply_damage` (line 139).
**Inputs:** `&mut PhysicsWorld`, attacker body id, `from`/`to` `Vec3`, base damage, `DamageType`, `&mut [Combatant]`.
**Outputs:** `Option<HitResult>`.
**Notes:** The `_radius` parameter is accepted but unused (the sweep is a pure raycast, not a shape-cast — see the docstring at lines 31-34 and the leading `_`). Stops at first hit (no pierce).

#### Stage detail — Path B
**File:** `astraweave-gameplay/src/ecs.rs:89-135` (`combat_system`), plugin at `ecs.rs:188-194`.
**Role:** ECS system registered into the `"simulation"` stage. Collects `(CAttackState, CTarget)` via `Query2`, and for active attacks within the combo step's timing window `[window.0, window.1]` and Manhattan reach, decrements target `CHealth.hp` by `step.damage`, advances `idx`, and deactivates at end of chain.
**Inputs:** ECS `World` with `CAttackState`, `CTarget`, `CPos`, `CInputState`, `CHealth`, and an `f32` dt resource.
**Outputs:** Mutated `CHealth` and `CAttackState`.
**Notes:** Distance is **Manhattan** (`|dx|+|dy|`, line 111), not Euclidean. `dt` defaults to `0.016` if no `f32` resource present (line 90).

### 2.B Weave op application (in-crate glue)

```text
caller (example/test): apply_weave_op(w, phys, nav_src, &mut budget, &op, &mut log)
    │
    ▼
weaving.rs:11
  role: match op.kind → either build a core DirectorPlan (Fortify/Collapse)
        or mutate PhysicsWorld directly (set_wind / clear_water), decrementing
        the matching WeaveBudget field; FreezeWater is presentation-only (no truth)
    │  apply_director_plan(w, &mut core_budget, &plan, log)  [core::validation:400]
    ▼
World terrain mutated (for terrain ops) and/or PhysicsWorld wind/water mutated
    │
    ▼
Ok(WeaveConsequence{drop_multiplier, faction_disposition, weather_shift})
```

#### Stage detail
**File:** `astraweave-gameplay/src/weaving.rs:11-150`
**Role:** Translates a positional `WeaveOp` into either a `astraweave_core::DirectorPlan` (`ReinforcePath`/`RaisePlatform` → `DirectorOp::Fortify`; `CollapseBridge` → `DirectorOp::Collapse`) applied via `apply_director_plan`, or a direct physics mutation (`RedirectWind` → `phys.set_wind`; `LowerWater` → `phys.clear_water`). Each arm bails if the relevant `WeaveBudget` field (`terrain_edits` or `weather_ops`) is `<= 0`, else decrements it by 1. `FreezeWater` (lines 94-105) is **presentation-only**: it consumes `weather_ops` and carries `op.a` for a render layer but makes **no** truth mutation (documented inline).
**Inputs:** `&mut World`, `&mut PhysicsWorld`, `&[Triangle]` nav source, `&mut WeaveBudget`, `&WeaveOp`, log closure.
**Outputs:** `Result<WeaveConsequence>`.
**Notes:** Re-bakes a throwaway navmesh each call (`NavMesh::bake(nav_src, 0.5, 55.0)` → `_nav`, line 113) that is immediately discarded — a demo-pathing artifact. The actual W-series "weave gameplay producer" that turns these ops into render-side `WeaveInstance`s lives in `examples/weaving_playground/src/weave_producer.rs` (commit `3edf15e25`), **not** in this crate.

### 2.C Veilweaver slice metadata → tutorial state

```text
WorldPartition cells (astraweave_scene::world_partition::Cell)
    │ VeilweaverSliceMetadata::from_cells(&[&Cell])   [veilweaver_slice.rs:108]
    ▼
veilweaver_slice.rs:80-114
  role: parse typed component views (WeaveAnchor, TriggerZone, DecisionPrompt,
        EncounterTrigger/Complete, EffectAnchor, SpawnPoint) into spec vecs
    │ WeaveTutorialState::from_metadata(meta)          [veilweaver_tutorial.rs:44]
    ▼
veilweaver_tutorial.rs:24-90
  role: index specs by id into HashMaps; hold mutable tutorial progress
    │ tutorial_anchor_sync / tutorial_trigger_system / tutorial_anchor_events
    ▼ (ECS systems consuming Events<TriggerVolumeEvent>, Events<AnchorStabilizedEvent>)
WeaveTutorialState mutated (anchors stabilized, triggers active)
```

#### Stage detail
**Files:** `astraweave-gameplay/src/veilweaver_slice.rs` (parsing) and `veilweaver_tutorial.rs` (state machine + 3 ECS systems at lines 122/136/157).
**Role:** Loads designer-authored slice content from `WorldPartition` cells into typed specs, then drives a tutorial progression state machine via ECS event drains.
**Inputs:** `Cell` component views (`CellComponentView`), ECS `Events<TriggerVolumeEvent>` / `Events<AnchorStabilizedEvent>`.
**Outputs:** `VeilweaverSliceMetadata`, mutable `WeaveTutorialState` resource.
**Notes:** The production driver is the `veilweaver_slice_runtime` crate (workspace root, not `astraweave-`-namespaced), which imports `tutorial_anchor_events`, `tutorial_anchor_sync`, `tutorial_trigger_system`, `WeaveTutorialState`, `VeilweaverSliceMetadata`, `WeaveAnchorSpec`, `TriggerZoneSpec` (`veilweaver_slice_runtime/src/lib.rs`), and the `veilweaver_demo` / `veilweaver_slice_loader` examples. The **bulk** of Veilweaver-slice gameplay logic (patterns, intents, anchors, abilities, combat, quests, echo currency) lives in the **separate `astraweave-weaving` crate**, not here — see §6.

## 3. Semantic Vocabulary

| Term | Definition | Used in |
|---|---|---|
| **Weave op** | A positional fate-weaving action (`WeaveOpKind`: ReinforcePath / CollapseBridge / RedirectWind / LowerWater / RaisePlatform / FreezeWater) with `a`, optional `b`, and `budget_cost` | `types.rs:40-61`, `weaving.rs` |
| **Weave budget** | Two integer pools (`terrain_edits`, `weather_ops`) consumed by weave ops | `types.rs:63-67`, `weaving.rs` |
| **Weave consequence** | The world-level after-effect of a weave op (drop multiplier, faction disposition, weather shift string) | `types.rs:33-38`, `weaving.rs` |
| **Combatant** | Physics-path combat participant: `{ body: u64, stats, iframes, parry }` | `combat_physics.rs:17-23` |
| **Combo chain / step** | ECS-path combo definition: ordered `ComboStep`s with `window`, `damage`, `reach`, `stagger` | `combat.rs:13-26`, `ecs.rs` (via `CAttackState`) |
| **Parry / IFrame** | Damage-gating windows: parry is consumed on hit (one-shot); iframes block without being consumed | `combat_physics.rs:6-15,115-136` |
| **Water movement mode** | Submersion-driven movement category (Dry/Wading/WaistDeep/Swimming/Diving), each with speed/stamina/oxygen multipliers | `water_movement.rs:12-61` |
| **Wet status** | Persistent debuff level (Dry/Damp/Wet/Soaking) affecting stamina regen/max | `water_movement.rs:66-108` |
| **Submersion** | Scalar `[0,1]` input to `WaterPlayerState::update` — supplied **by the caller**, not computed from any water field | `water_movement.rs:206` |
| **Weave anchor** | A Veilweaver loom node the player stabilizes; spec parsed from a cell `WeaveAnchor` component | `veilweaver_slice.rs:6-14`, `veilweaver_tutorial.rs` |
| **Trigger zone / decision prompt / encounter trigger** | Designer-authored cell volumes/prompts that drive slice progression | `veilweaver_slice.rs`, `veilweaver_tutorial.rs` |

### Terms to NOT confuse

- **In-crate `weaving.rs` weave-op glue vs the standalone `astraweave-weaving` crate.** This crate's `WeaveOp`/`WeaveBudget`/`WeaveConsequence`/`apply_weave_op` are a thin Director-plan/physics bridge. The separate `astraweave-weaving` crate is a large emergent-behavior system (`PatternDetector`, `IntentProposer`, `WeaveAdjudicator`, `Anchor`, `EchoCurrency`, its own `Quest`/`CombatSystem`). They share the *word* "weave"/"Veilweaver" and some concept names (Quest, Anchor, combat) but are different crates with different types. See §6.
- **`water_movement.rs` (player swimming/oxygen logic) vs the W-series render-side water.** `water_movement.rs` is CPU player-mechanic logic that takes a scalar `submersion` and outputs speed/stamina/oxygen multipliers. The W-series "Water Successor" render-side deformation (`astraweave-render` `water.rs`/`water.wgsl`, fed by `Renderer::set_water_weave_instances`) is unrelated and lives in the render crate; the two never connect.
- **Combat Path A (`perform_attack_sweep`, physics) vs Path B (`combat_system`, ECS combo).** Two unrelated combat implementations in one crate.
- **`Stats` (this crate, `stats.rs`) vs `CompanionState`/`CHealth` (core).** Combat Path A uses gameplay `Stats`; Path B uses core `CHealth`.

### Naming collisions

- **"Weave" / "Veilweaver"**: In `astraweave-gameplay`, refers to positional `WeaveOp`s (terrain/weather edits) and the slice tutorial metadata. In `astraweave-weaving`, refers to a pattern→intent→adjudication emergent-behavior layer plus the anchor/echo ability system. Same vocabulary, different crates and types.
- **"Quest"**: `astraweave-gameplay::quests::Quest` (objectives/tasks for the UI/HUD) vs `astraweave-weaving::quest::Quest` (`QuestManager`, `ObjectiveType`, `QuestReward`). Distinct types in distinct crates.
- **"Combat"**: `combat.rs`/`combat_physics.rs`/`ecs.rs::combat_system` here vs `astraweave-weaving::combat::CombatSystem`. Three+ combat surfaces across the two crates.
- **"Anchor"**: `WeaveAnchorSpec`/`AnchorStatus` here (slice tutorial loom nodes) vs `astraweave-weaving::anchor::Anchor` (`AbilityType`, `AnchorVfxState`). Related design concept, different implementations.
- **"Water"**: `water_movement.rs` (player swim/oxygen logic) vs the W-series render water (`astraweave-render`). No code relationship.

### Known cognitive traps

- **Trap**: Assuming `water_movement.rs` is wired into the W-series water system or consumes a `WaterQuery`/fluids facade.
  **What's actually true**: It is standalone CPU logic taking a scalar `submersion` parameter (`water_movement.rs:206`); the crate has no `astraweave-fluids` dependency; and it has zero workspace callers. It is dormant residue from the abandoned plan in `WATER_SYSTEM_ENHANCEMENT_PLAN.md` (which carries a staleness banner). The render-side water successor is in `astraweave-render`.
- **Trap**: Treating the gameplay-crate `weaving.rs` as the Veilweaver weaving system.
  **What's actually true**: The substantive Veilweaver gameplay logic is the separate `astraweave-weaving` crate. The gameplay-crate `weaving.rs` is a small Director-plan/physics bridge; the W-series render producer (`WaterWeaveProducer`) is in `examples/weaving_playground/`.
- **Trap**: Assuming `perform_attack_sweep`, `apply_weave_op`, or the ECS combat/craft/quest plugins are wired into an engine runtime.
  **What's actually true**: their only non-test drivers are `examples/*` binaries. No `astraweave-*` engine crate calls them.
- **Trap**: Reading combat in `ecs.rs` and combat in `combat_physics.rs` as one system.
  **What's actually true**: they are two independent implementations (Manhattan ECS combo vs Rapier raycast sweep) with no shared code.

## Appendix A: Quick reference for agents

**If you're working on this system, remember:**
1. **This crate is a grab-bag, not a pipeline.** Edit the one module you need; don't assume modules interconnect.
2. **The headline behavioral functions are example-driven.** `perform_attack_sweep`, `apply_weave_op`, and the ECS combat/craft/quest plugins have no in-engine production caller. Adding a new one is a real integration step, not a no-op.
3. **`water_movement.rs` is dormant residue** from an abandoned plan — and it is NOT the W-series render water. Don't wire new water gameplay through it without first resolving §11.
4. **"Weave"/"Veilweaver"/"Quest"/"Combat"/"Anchor" each name two different things** across this crate and `astraweave-weaving`. Always qualify the crate.
5. **`FreezeWater` is presentation-only** (no truth mutation) by deliberate decision (§7).

**Files you'll most likely touch:**
- `astraweave-gameplay/src/combat_physics.rs` (physics combat)
- `astraweave-gameplay/src/ecs.rs` (ECS combat/craft/quest components + systems)
- `astraweave-gameplay/src/weaving.rs` + `types.rs` (weave ops)
- `astraweave-gameplay/src/veilweaver_slice.rs` / `veilweaver_tutorial.rs` (slice content)

**Files you should NOT touch without strong reason:**
- `astraweave-gameplay/src/water_movement.rs` — dormant; resolve §11 before extending.
- `astraweave-gameplay/src/weave_portals.rs` / `weave_telemetry.rs` — wiring unverified; confirm a caller before building on them.

**Common mistakes when changing this system:**
- **Mistake**: Adding a feature to this crate's `weaving.rs` assuming it is the Veilweaver weaving system.
  **Why wrong**: The substantive weaving system is the separate `astraweave-weaving` crate; this is a small Director-plan bridge.
- **Mistake**: Wiring `water_movement.rs` into the W-series water render path.
  **Why wrong**: They are unrelated; `water_movement` is dormant CPU logic with no fluids dependency.
- **Mistake**: Marking a task complete because tests pass.
  **Why wrong (per CLAUDE.md Key Lesson 8)**: Several modules here are tested-but-dormant. "Wired beats tested" — confirm a production caller.

## Appendix B: Historical context

`astraweave-gameplay` accreted as the gameplay-logic layer for the AstraWeave demos and the Veilweaver vertical slice. Two campaign threads are visible in its recent history:

1. **The Unified Camera campaign (C.7.A)** rewired `cutscenes.rs` to emit `astraweave_cinematics::CameraKey` and added the `astraweave-cinematics` dependency (commit `91eda3139`, Cargo.toml comment).
2. **The W-series "Water Successor" campaign** added the `FreezeWater` weave op as a presentation-only variant (commit `3edf15e25`) while the broader water gameplay direction (swimming/oxygen, embodied in `water_movement.rs`) was abandoned in W.1 in favor of a render-only layered water system (`WATER_SYSTEM_ENHANCEMENT_PLAN.md` staleness banner; recovery tag `w0-pre-deprecation`).

The Veilweaver slice itself spans three places: the gameplay-crate's `veilweaver_slice.rs`/`veilweaver_tutorial.rs` (cell metadata + tutorial state), the separate `astraweave-weaving` crate (emergent-behavior gameplay logic), and the `veilweaver_slice_runtime` crate (the headless 30-minute-demo harness). This trace covers only the gameplay-crate portion; the other two are separate subsystems.
